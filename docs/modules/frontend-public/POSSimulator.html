POS Simulator Module

Interactive financial product simulator allowing customers to compare online and POS payment options, with real-time amortization calculations and fallback to local computation for static deployments.

Module Overview

The POS Simulator module provides a self-contained simulation experience for special banking products. Users can input a product, loan amount, term, and customer type to immediately see:

  • Monthly payment calculated via standard amortization formula
  • Total cost including commission and insurance fees
  • Breakdown of interest, commission, and insurance
  • Online payment equivalent (2.5% surcharge)
  • POS payment equivalent (3.5% surcharge, individual customers only)
  • Maximum installments for POS (up to 12 months)

Technical Foundation

  • Frontend: React 18 + React Router SPA page component
  • Backend: ASP.NET Core POST endpoint at /api/pos-simulator/calculate
  • Localization: i18next with full es/en/pt translation keys under posSimulator.*
  • Fallback: Automatic client-side calculation when backend unavailable
  • State: React hooks (useState, useEffect) with form validation

Data Flows

Happy Path: Successful Calculation

  1. User fills form: product ID, amount, term, customer type
  2. Client validates form (non-empty product, amount > 0, term > 0)
  3. Client attempts POST to /api/pos-simulator/calculate with payload
  4. Backend calculates amortization and returns POSSimulationResult
  5. Client validates response has required fields (monthlyPayment, etc.)
  6. Results displayed in summary cards, breakdown table, and payment method cards

Fallback: Backend Unavailable

  1. Backend POST fails or returns invalid response (e.g., HTML from Surge)
  2. Client catches error and calls calculateLocally()
  3. Local calculation uses product's baseInterestRate or defaults
  4. Same results structure returned to UI, seamless experience

Validation Failures

  1. User submits form with missing/invalid data
  2. Client shows inline error message from i18n key
  3. User corrects input and resubmits

Backend Endpoint

Endpoint: POST /api/pos-simulator/calculate

Authorization: [AllowAnonymous] — public endpoint

Request Body (POSSimulationRequest):

  • productId (string, required)
  • amount (decimal, required, > 0)
  • termMonths (int, required, 1–60)
  • customerType (string, default "individual")

Response Body (POSSimulationResult):

  • amount (decimal)
  • termMonths (int)
  • interestRate (double, as %)
  • commission (decimal)
  • insurance (decimal)
  • monthlyPayment (double)
  • totalCost (double)
  • totalInterest (double)
  • onlinePaymentAvailable (bool)
  • onlinePaymentEquivalent (double)
  • posPaymentAvailable (bool)
  • posPaymentEquivalent (double?)
  • posInstallments (int?)
  • calculatedAt (DateTime)

Dependencies and Services

  • creditTypeService: Loads product list with fallback to mock data (mandatory)
  • axios: HTTP client for backend POST (wrapped with error handling)
  • react-router-dom: Navigation and routing
  • react-i18next: Translations
  • lucide-react: ChevronDown icon for select dropdowns

Deployment Constraints (Critical)

  • Static SPA on Surge: Every unknown URL returns 200 with index.html, not 404. This means axios must validate response before using it.
  • No raw axios.get for products: Always use creditTypeService which has fallback to mock data.
  • Local calculation must work: If backend is down, the simulator must still function with client-side amortization.
  • Array guard: Always use Array.isArray(data) ? data : [] before calling .map() on products.
