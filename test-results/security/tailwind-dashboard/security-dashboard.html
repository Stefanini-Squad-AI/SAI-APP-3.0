Resumen de Evaluación de Seguridad OWASP ZAP: TuCreditoOnline (TCO)

Postura General El escaneo OWASP ZAP del frontend y API de TCO en entorno de desarrollo/staging muestra una postura de seguridad de bajo riesgo con 8 tipos de hallazgos únicos en 133 instancias, todos de severidad Baja o Informativa—sin riesgos Altos o Medios. Indica protecciones sólidas contra amenazas mayores, pero oportunidades para fortalecer headers y prácticas de caché antes de producción.[1][2]

Temas Principales

Brechas en Headers de Seguridad: Tres hallazgos Baja/Informativos señalan headers ausentes o inválidos (Cross-Origin-Resource-Policy, X-Content-Type-Options), que podrían exponer usuarios a ataques de sniffing o cross-origin en producción.[1][3]

Manejo de Caché y Contenido: Alertas Informativas múltiples sobre contenido almacenable/cacheable, contenido no almacenable y Content-Type inesperado indican políticas de caché inconsistentes, arriesgando exposición de datos sensibles.[1]

Ruido Operacional: Errores de cliente altos (110 instancias) y solicitudes de autenticación (3) son típicos en dev, pero revisar para no enmascarar issues reales.[2][4]

Priorización Priorizar primero los 3 hallazgos de Baja severidad (11 instancias), por impacto potencial en negocio como fugas de datos. Items Informativos (122) son menor prioridad, pero monitorear. Sin bloqueadores inmediatos para prod.[1][5]

    Recomendaciones Accionables
  • Auditar e Implementar Headers: Agregar X-Content-Type-Options: nosniff, Cross-Origin-Resource-Policy: same-origin en 2 semanas.
  • Estandarizar Políticas de Caché: Enforzar Cache-Control: no-store en endpoints sensibles; re-escaneo ZAP.
  • Reducir Errores de Cliente: Investigar top errores; integrar ZAP en CI/CD.[2][3]
  • Establecer Baselines: Escaneos pasivos semanales en staging.[4]
  • Capacitación: Entrenar devs en interpretación ZAP y mejores prácticas.

Description

Cross-Origin-Resource-Policy header is an opt-in header designed to counter side-channels attacks like Spectre. Resource should be specifically set as shareable amongst different origins.

Solution

Ensure that the application/web server sets the Cross-Origin-Resource-Policy header appropriately, and that it sets the Cross-Origin-Resource-Policy header to 'same-origin' for all web pages. 'same-site' is considered as less secured and should be avoided. If resources must be shared, set the header to 'cross-origin'. If possible, ensure that the end user uses a standards-compliant and modern web browser that supports the Cross-Origin-Resource-Policy header (https://caniuse.com/mdn-http_headers_cross-origin-resource-policy).

Description

The Anti-MIME-Sniffing header X-Content-Type-Options was not set to 'nosniff'. This allows older versions of Internet Explorer and Chrome to perform MIME-sniffing on the response body, potentially causing the response body to be interpreted and displayed as a content type other than the declared content type. Current (early 2014) and legacy versions of Firefox will use the declared content type (if one is set), rather than performing MIME-sniffing.

Solution

Ensure that the application/web server sets the Content-Type header appropriately, and that it sets the X-Content-Type-Options header to 'nosniff' for all web pages. If possible, ensure that the end user uses a standards-compliant and modern web browser that does not perform MIME-sniffing at all, or that can be directed by the web application/web server to not perform MIME-sniffing.

Description

The response contents are storable by caching components such as proxy servers, and may be retrieved directly from the cache, rather than from the origin server by the caching servers, in response to similar requests from other users. If the response data is sensitive, personal or user-specific, this may result in sensitive information being leaked. In some cases, this may even result in a user gaining complete control of the session of another user, depending on the configuration of the caching components in use in their environment. This is primarily an issue where "shared" caching servers such as "proxy" caches are configured on the local network. This configuration is typically found in corporate or educational environments, for instance.

Solution

Validate that the response does not contain sensitive, personal or user-specific information. If it does, consider the use of the following HTTP response headers, to limit, or prevent the content being stored and retrieved from the cache by another user: Cache-Control: no-cache, no-store, must-revalidate, private Pragma: no-cache Expires: 0 This configuration directs both HTTP 1.0 and HTTP 1.1 compliant caching servers to not store the response, and to not retrieve the response (without validation) from the cache, in response to a similar request.
